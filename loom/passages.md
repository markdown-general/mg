## Example 9 (buff, layered depth and specialization)

**Selected passage:**
"When you've grounded yourself in card, yin, flow, and deck, buff is where depth lives. Buff contains specialist patterns: haskell-checklist.md (recurring work), session.md (immutable conversation trees), lattice.md (paired comparative elaboration), drafting.md (how to write cards), range.md (casting technique), duality.md (holding dual states), engineering.md (queue architecture, idempotent agents), loop.md (humans in/out decision), literary.md (philosophical stance), and domain work (browser patterns, emacs integration, pi-mono)."

**Graph relationships traced:**
- card → elaborate (buff extends elaborate through specialist patterns)
- card → encoding (drafting teaches how to encode strategy)
- flow → path (recurring checklists make permanent flow explicit)
- flow → context (sessions create immutable conversation trees)
- compose → intent (duality and range enable deeper intent encoding)
- collaborate → kernel (engineering makes agents idempotent)
- collaborate → multiplicity (loop enables humans and agents to coordinate decisions)
- agent → semantics (domain patterns add specialist semantic depth)
- elaborate → context (session branching and compaction organize context)

**Paraphrases:**
- buff is depth that comes after foundation
- foundation = card, yin, flow, deck mastery
- buff = specialist knowledge and capability upgrades
- recurring work crystallizes as permanent flow
- sessions create branching conversation trees
- lattice structures enable paired elaboration
- agents are made idempotent through engineering
- humans stay in the loop for critical decisions
- domain patterns (haskell, emacs, browser, pi-mono) are semantic depth
- philosophical stance (literary) shapes how work is framed

**Buff note:**
Buff represents the layer that emerges *after* a maker has internalized the core operations (card, yin, flow, deck). Where the core teaches *how to coordinate*, buff teaches *what depths are possible*. A recurring checklist (haskell-checklist) is permanent flow—it crystallizes patterns you've executed repeatedly and encodes them as standing instructions. Sessions create immutable conversation trees with branching and forking—the topology of thought preserved. Lattices enable paired comparative elaboration, where two related concepts are held in tension and their relationship clarified. Engineering ensures agents can be idempotent: run the same card ten times with the same inputs and get the same output, making agents reliable participants. Loop enables the human to stay in the loop at decision points, preventing over-delegation. Buff is where the system gains both *specialization* (domain-specific patterns) and *recursion* (humans and agents can descend deeper into any domain without losing coordination). The phrase "when you've grounded yourself" suggests buff requires fluency in the foundation; you don't jump to buff, you grow into it. It's how makers move from operational competence to strategic depth, where every domain—haskell, emacs, browser development, pi-mono—can be explored at the depth it deserves while remaining integrated into the larger coordination structure.

**Selected passage:**
"Cards describe strategy — what we intend to do. But strategy also has shape: actions, decision points, jumps back and forward, uncertainty, failure. When work is in motion you need to track position and topology quickly. Flow encoding marks these structures so the card shows where you are and what's possible next."

**Graph relationships traced:**
- card → intent (cards encode strategy)
- intent → flow (strategy has shape and topology)
- flow → path (branching, jumps, decisions)
- flow → elab (marking uncertainty and failures)
- flow → context (position marking: "where you are now")
- card → flow (flow lives inside card)
- compose → intent → flow → path (intent moves through elaboration to executable paths)

**Paraphrases:**
- cards encode strategy
- strategy has shape and topology
- flow marks actions and decisions
- flow marks jumps back and forward
- flow marks uncertainty and failure
- position marks show where execution is now
- flow marks make topology scannable
- symbols encode structure within prose

**Multiplicity note:**
Flow is the grammar that turns abstract intent into executable topology. A card contains strategy (what we intend to do), but strategy only becomes actionable when given shape: the decision points where paths fork, the failure markers where plan met reality, the position marks showing where focus is now. Flow symbols (⊢, 🚩, [?], ◊, ⟜, ¹) are not decoration—they're a compression of intent into topology. They let yin track position and branching in real time without breaking prose narrative. The symbols work because they mark structure without imposing it; if topology gets unclear, prose carries the meaning. This is why flow lives *inside* cards rather than as a separate tracking layer: the shape of work and the record of work are unified on the same surface. Intent flows into flow marks, which articulate the paths available next. The card becomes both what was intended and a map of where the work can go.

**Selected passage:**
"**compression: prose → deck** Find leads hiding in prose; Strip rubbish tokens; Surface elaboration patterns; Assign dash types. **expansion: deck → prose** Leads become sentences; Elaborations flow naturally; Structure dissolves but remains; Dash types become prose connectors. The grammar makes visible what was already there."

**Graph relationships traced:**
- prose ↔ deck (bidirectional relationship; structure and content are not separate)
- prose → syntax (leads, tokens, elaborations as structural elements)
- deck → semantics (dash types carry meaning; elaboration patterns emerge)
- syntax → semantics (dashes type relationships; prose flows naturally)
- card → deck (cards are composed of decks)
- token → lead → slug → deck → card (compositional chain across scales)
- agent → card (agents read/compose at all levels of this chain)

**Paraphrases:**
- prose hides structure inside narration
- compression surfaces hidden leads
- decks make elaboration patterns visible
- dashes type the relationships
- expansion dissolves structure back to prose
- structure was already there
- grammar names what's hidden
- tokens ↔ leads ↔ slugs ↔ decks ↔ cards (all continuous)

**Multiplicity note:**
Decking reveals the core insight that structure and prose are not antagonistic but aspects of the same thing seen at different scales. Compression doesn't impose structure—it *discovers* structure that narration already contains. A comma is already a dash. A preposition already types a relationship. When you deck, you're naming the grammar already operating in what you wrote. This enables the bidirectional flow: humans write prose carrying implicit structure; agents can read structure and regenerate prose; both are working the same surface because "the grammar makes visible what was already there." The compositional chain (tokens ↔ leads ↔ slugs ↔ decks ↔ cards) means you can zoom between scales—compress prose to deck for clarity, expand deck back to prose for flow—without losing information. Structure is not lockdown; it's illumination of patterns that natural language always carried. This is why deck works across the commons: it respects prose as primary, makes structure visible when needed, and lets either human or agent move between compressed and expanded forms without translation.

## Example 8 (drift, semantic state misalignment)

**Selected passage:**
"Drift happens when understanding (notes-state) diverges from reality (file-state). Patterns evolve. Systems operate. Meaning shifts. This is structural, not accidental... The most common mistake: agents inherit old flow marks that were already completed and treat them as new mandates. They don't distinguish between work that was done and work that needs doing. When you read a card with flow marks, ask: are these active scaffolding or historical record? Has this path been walked? Is the branching still live?"

**Graph relationships traced:**
- card → semantics (notes-state: what we think the card says)
- card → encoding (file-state: what the card actually is)
- semantics ≠ encoding (drift occurs when these diverge)
- agent → card (agents read cards and inherit their state)
- flow → path (flow marks encode completed vs active paths)
- flow → context (old flow marks persist in context)
- card → meaning (divergence between understanding and reality)
- elaborate → context (when agents elaborate stale cards, context drifts)
- agent → semantics (agents misinterpret semantic state of old marks)

**Paraphrases:**
- understanding diverges from reality
- patterns evolve; meaning shifts
- agents inherit old flow marks
- agents mistake completed work for active work
- flow marks hide historical record in active scaffolding
- cards become ambiguous: what was done vs what needs doing
- drift is structural, not accidental
- context accumulates stale instructions
- semantic state must be explicitly reconciled

**Drift note:**
Drift exposes the tension between card-as-artifact (persisting in files, accumulating history) and card-as-active-instruction (what needs doing now). Flow marks were designed to be scannable and concise, but this compression creates ambiguity over time: does `⊢ fix all the warnings` mean "fix these warnings (done)" or "go fix all warnings (pending)"? Agents, operating without full historical context, cannot reliably distinguish. The semantic state of a card—what parts are live scaffolding versus completed historical record—is not encoded in the card itself; it lives in the implicit understanding between collaborators. This is why drift detection requires human judgment: only a coordinator who knows the actual execution history can reconcile notes-state against file-state and determine which flow marks are atemporal records and which ones are still active mandates. This is not failure; it's the cost of asynchrony. Cards persist and travel across time and agents, accumulating meaning. The system handles this by making drift visible (🚩 marks where plan met reality) and forcing conscious choice: rewind, replan, or start fresh. The alternative—perfect semantic consistency across distributed asynchronous execution—is incompatible with the generative, evolving nature of the work.

## Example 10 (yin, the coordination rhythm)

**Selected passage:**
"**yin** ⟜ coordinator at the center of the flow. **yin** ⟜ executing strategy through a cycle. **Flow** ⟜ Check state: is intent clear? What's next? **Act** ⟜ Commit to a card and execute; decision made. **Breathe** ⟜ Review what happened; sync on signals; update flow marks. When work is in motion, coordination requires staying at the center. Heavy IO doesn't happen in your head—you spin agents to handle it. Keep your context lean."

**Graph relationships traced:**
- compose → intent (check state: is intent clear?)
- intent → flow (what's next? decision points)
- flow → path (commit to a card, path is chosen)
- card → encoding (execute the committed decision)
- elaborate → context (review what happened)
- flow → elab (sync on signals; update flow marks)
- collaborate → agent (spin agents for heavy IO)
- agent → elaborate (agents work while yin stays centered)
- context → path (lean context enables quick decisioning)
- card → semantics (intent must be clear to act)

**Paraphrases:**
- yin is the coordinator at center
- three-part cycle: flow, act, breathe
- flow checks intent clarity
- act commits to card
- breathe reviews and syncs
- agents handle IO work
- coordinator keeps context lean
- coordination happens in real time
- intent flows through the cycle
- signals update flow marks

**Yin note:**
Yin is not a role that does all the work—it's a *position* at the center of coordination. The three-part cycle (Flow → Act → Breathe) is a rhythm that keeps a human coordinator grounded while agents handle external work. **Flow** is the checkpoint: is the intent clear? Do we know what's next? This is where ambiguity surfaces and gets resolved before committing. **Act** collapses uncertainty into decision: you choose a card and execute it. The decision is the commit; the card is the execution path. **Breathe** is the review: what actually happened? Did the reality match intent? This is where flow marks get updated, where signals from agents get synced, where semantic state gets refreshed. The genius is that a human can stay *centered*—cognitively present—by staying out of the heavy lifting. Heavy IO doesn't belong in yin's head; that's what agents are for. You spin them to read files, write outputs, refine execution. Yin stays lean, focused on three things: checking intent, making decisions, and reviewing reality. The cycle is synchronous (yin drives it), but the execution is asynchronous (agents do the work). This is how multiplicity is managed without multiplying the coordinator's cognitive load. By keeping the center tight and the periphery loose—agents handle elaboration, file IO, refinement—yin can stay present for the decisions that matter: is this what we intended? What comes next? Did reality match the plan?
