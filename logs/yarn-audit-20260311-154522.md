## ⚠ ISSUES REQUIRING CLARIFICATION

### HypH.hs — **Speculative Reasoning Without Verification**

**Issue 1: "Why Arrow?" section is intuitive but incomplete**
```haskell
-- In @Hyp@, @⊙@ is:
-- f ⊙ g = Hyp $ \h -> ι f (g ⊙ h)
-- The @\h ->@ is free because @arr = (->)@.
-- For @HypH arr@, the corecursion @(g `zipper`)@ is a Haskell function that
-- needs lifting into @arr@. That requires @arr (g `zipper`)@, i.e. @Arrow arr@.
```
**Problem**: Claims "needs lifting" but doesn't explain **why** zipper is different from @(->)@.
- The step is: zipper is composition, composition for HypH needs to lift the function into arr.
- For arr = (->), lifting is free (just apply).
- For arr = Arrow, lifting requires arr to be an arrow.
- **Current doc**: Implies this but doesn't explicitly state it.

**Action**: Clarify the distinction. Add a concrete example showing @zipper :: HypH arr c a -> HypH arr b a@ (a function) that must become @arr (zipper)@ (a morphism in arr).

**Issue 2: "The Rec {} problem" — Optimization Claim Without Benchmarks**
```haskell
-- @Traced MealyM@ compiles via a recursive interpreter in @Rec {}@.
-- Inlining is blocked; composed state is a heap tuple per byte.
-- @HypH MealyM@ encodes the same structure corecursively. @zipper@ is
-- productive — each call unfolds one @ι f@ layer before recurring.
-- GHC sees plain @MealyM@ compositions with no recursive interpreter.
```
**Problem**: Presents as fact but:
- No performance numbers provided
- "GHC sees plain @MealyM@ compositions" — does it really? Under what conditions?
- "heap tuple per byte" — is this measured or theoretical?

**Action**: Either:
1. Mark as hypothesis: "Hypothesis: @HypH MealyM@ may avoid Rec{} inlining issues" + cite what would verify
2. Provide benchmark: show Core output or timing data
3. Remove: if unverified, move to a design rationale section with caveats

```
**Problem**: Doesn't answer:

- What's the performance trade-off?
- When is zipper better than recursive interpreter?

**Action**: Add to module doc:
```
= Usage Patterns

Use @toHypH@ when:
- You have a stateful Mealy machine and want corecursive unfolding
- You need to avoid Rec{} in Core output (GHC inlining issues)
- You're composing many Mealy machines (zipper unfolds lazily)

Use Traced Mealy when:
- Simplicity is more important than optimization
- Your machines are small (inlining doesn't matter)
- You don't have performance bottlenecks
```

Reference: Boccali et al., "Bicategories of Automata", Section X.
```
