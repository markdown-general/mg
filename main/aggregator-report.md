## 4. Recommendation: Probable Design Intent

If I had to bet:

### **The Design is:**

**Session Write Model**: Agents write directly to branched sessions (Q4 C2 + validation from C3)
- Agents append entries to their forked branch
- Memo validates afterward (post-hoc checks: parent chain, depth, consistency)
- Yin doesn't block; memo enforces invariants

**Agent Lifecycle**: Seasonal, with elders (Q9 C3)
- Each card/phase spawns fresh agent
- Agent accumulates state in grepl (Q13 C3) or checkpoints to session (CompactionEntry)
- At 60% or phase end: agent becomes "elder" (immutable reference, later agents learn from it)
- This explains "responsible citizen" arc: agents internalize context fullness, not as external signal

**Memo**: Full SessionManager (Q10 C2)
- Reimplements pi-mono's session ops in Haskell
- Owns branching, drift validation, elder thresholds, compaction
- Built on yarn (which provides parsing + tree traversal)
- Tightly coupled to pi-mono schema but batteries-included

**Yarn**: Lexer + Parser + Tree Ops (Q5 C2)
- Fast JSON parsing (tcat core)
- Generic tree traversal (parentId-based, schema-agnostic)
- Publishable independently on Hackage
- Memo builds on it (yarn is the fast layer, memo adds session semantics)

**Landing Order**: Parallel with checkpoints (Q18 C3)
- Week 1: Yarn lexer+parser β. Memo architecture designed. PoC plan locked.
- Week 2-3: Yarn on Hackage. Memo parses real pi-mono sessions. First agent spins.
- Week 4: Full stack validated. Backward pass (Q11 C3, Q16 C3) works. Tools finalized.
- Why? Document's exploratory tone suggests you're learning as you go. Parallel + checkpoints = safe iteration.

**PoC**: Agent branching (Q19 C2, not C1)
- Proves full cycle: session fork → agent writes → branch created → result integrated
- Doesn't require backward-pass yet (Q19 C3 is future), but validates the core spinning loop
- Uses real SessionManager, real branching, real tree mutations

**Tools**: SDK library + thin CLI (Q20 C3)
- Core: `yarn-agent` Haskell library with SessionManager, SpinControl, TreeQuery
- CLI: thin wrapper (`yarn-agent spin`, `yarn-agent list`, etc.)
- Extensibility: users can import library, write custom Haskell workflows
- Why? Library-first enables advanced use cases (e.g., grepl integration, backward-pass queries) without bloating CLI

**Drift Detection**: Post-hoc validation (Q15 C3)
- Agent executes freely
- After agent exits, yin validates trace closure (did dependencies stay consistent?)
- If drift: mark 🚩, preserve as historical record, optionally retry
- Why? Aligns with agent agency; doesn't block execution; backward pass can reason about divergences

**Slop Reduction**: Narrative coherence (Q14 C2)
- Agents see branched path only (not full history)
- Card provides explicit context
- Result: signal-to-noise improves 5:1 even if token count is same
- Why? Supports agent reasoning without requiring context truncation; cheaper models work better
