# session ⟜ Immutable, branching conversation records

Sessions are the core unit of work in pi-mono. They're append-only JSONL files that record every action: messages, tool execution, model changes, branching decisions, user annotations, and extension-specific data.

**Core philosophy**: Minimal entry types, maximum extensibility. All operations are append-only writes to a tree-structured log. Sessions are simultaneously operational records (live replay) and auditable artifacts (forensic analysis, reproducibility).

---

## Anatomy of a Session

A session file (JSONL, one entry per line):

```
{"type":"session","version":3,"id":"abc-123","timestamp":"2026-03-02T...","cwd":"/path/to/project","parentSession":"../other.jsonl"}
{"type":"message","id":"msg-1","parentId":null,"timestamp":"...","message":{"role":"user","content":[...]}}
{"type":"message","id":"msg-2","parentId":"msg-1","timestamp":"...","message":{"role":"assistant","content":[...]}}
{"type":"model_change","id":"model-1","parentId":"msg-2","timestamp":"...","provider":"anthropic","modelId":"claude-opus"}
{"type":"session_info","id":"meta-1","parentId":"model-1","timestamp":"...","name":"flip"}
{"type":"label","id":"label-1","parentId":"meta-1","timestamp":"...","targetId":"msg-2","label":"key insight"}
{"type":"branch_summary","id":"branch-1","parentId":"msg-2","timestamp":"...","summary":"Explored approach X, pivoting to Y"}
```

### Entry Types (Core)

**SessionHeader** ⟜ one per file, first entry
- `id` ⟜ session UUID
- `cwd` ⟜ working directory where session started
- `parentSession` ⟜ path to parent session (if forked), enables lineage
- `version` ⟜ format version (currently 3)

**SessionMessageEntry** ⟜ agent ⟷ user messages
- `message` ⟜ AgentMessage (role: user|assistant, content: text|tool_call|tool_result)
- Forms the linear conversation history

**ModelChangeEntry** ⟜ LLM model/provider switch
- `provider` ⟜ "anthropic", "openai", etc.
- `modelId` ⟜ full model identifier
- Allows tracking model evolution within session

**ThinkingLevelChangeEntry** ⟜ Extended thinking toggle
- `thinkingLevel` ⟜ "low", "medium", "high"
- Captures when reasoning depth changes

**SessionInfoEntry** ⟜ User-facing metadata
- `name` ⟜ Display name (e.g., "flip", "research-session-1")
- Set via `appendSessionInfo()` or TUI `Ctrl+R`
- Read by session browsers for sorting/filtering

**LabelEntry** ⟜ User annotations on specific entries
- `targetId` ⟜ ID of entry being labeled
- `label` ⟜ Arbitrary string ("key decision", "research", etc.)
- Enables navigation and filtering in tree browsers

**BranchSummaryEntry** ⟜ Marks branching points
- `fromId` ⟜ ID where branch starts
- `summary` ⟜ Human or AI-generated context ("explored approach X, pivoting Y")
- `fromHook` ⟜ True if generated by extension hook
- `details` ⟜ Extension-specific metadata (artifacts, diffs, etc.)

**CompactionEntry** ⟜ Compression waypoints
- `summary` ⟜ Summarized context from compressed entries
- `firstKeptEntryId` ⟜ First entry after compression window
- `tokensBefore` ⟜ Token count pre-compression
- `details` ⟜ Extension data (ArtifactIndex, version markers)
- `fromHook` ⟜ True if from extension compaction hook

**CustomEntry** ⟜ Extension-opaque data
- `customType` ⟜ Extension identifier (prevents collisions)
- `data` ⟜ Arbitrary JSON
- Does NOT participate in LLM context (metadata-only)

**CustomMessageEntry** ⟜ Extension messages in LLM context
- `customType` ⟜ Extension identifier
- `content` ⟜ String or TextContent|ImageContent array
- `details` ⟜ Extension metadata (not sent to LLM)
- `display` ⟜ True = render in TUI with special styling; false = hidden

All entries have: `id` (unique), `parentId` (tree structure), `timestamp` (ISO 8601).

## SessionManager API ⟜ The Core Harness

Located: `@mariozechner/pi-coding-agent/core/session-manager.ts`

### Read & Navigate

```typescript
// Get current state
getSessionId(): string
getSessionFile(): string | undefined
getCwd(): string
getHeader(): SessionHeader | null
getSessionName(): string | undefined
getLeafId(): string | null                    // Current position in tree
getLeafEntry(): SessionEntry | undefined

// Query entries
getEntry(id: string): SessionEntry | undefined
getChildren(parentId: string): SessionEntry[]
getBranch(fromId?: string): SessionEntry[]   // Path from root to specified leaf
getTree(): SessionTreeNode[]                  // Full tree structure with children
getLabel(id: string): string | undefined

// Get context (what gets sent to LLM)
getSessionContext(): SessionContext           // {messages, thinkingLevel, model}
getEntries(): SessionEntry[]                  // All entries in order
```

### Write ⟜ Append Operations

```typescript
// Conversation
appendMessage(message: Message | CustomMessage): string
  returns entry id

// Configuration
appendThinkingLevelChange(thinkingLevel: string): string
appendModelChange(provider: string, modelId: string): string
appendSessionInfo(name: string): string

// Annotation
appendLabelChange(targetId: string, label: string | undefined): string

// Extension
appendCustomEntry(customType: string, data?: unknown): string
appendCustomMessageEntry<T>(
  customType: string,
  content: string | (TextContent | ImageContent)[],
  display: boolean,
  details?: T
): string

// Compression
appendCompaction<T>(
  summary: string,
  firstKeptEntryId: string,
  tokensBefore: number,
  details?: T,
  fromHook?: boolean
): string
```

### Navigate & Branch ⟜ Tree Surgery

```typescript
// Move the "current position" (leaf pointer)
branch(branchFromId: string): void
  // Move leaf to entry; next append() creates new branch

branchWithSummary(
  branchFromId: string | null,
  summary: string,
  details?: unknown,
  fromHook?: boolean
): string
  // Like branch() but appends BranchSummaryEntry first

resetLeaf(): void
  // Move to root (before any entries)
  // Useful for re-editing the first user message

// Extract a branch to a new file
createBranchedSession(leafId: string): string | undefined
  // Writes a new session file containing only the path root → leafId
  // Useful for saving a specific conversation thread
  // Preserves parentSession link for lineage
```

### Fork & List ⟜ Multi-Project

```typescript
// Fork from another project into current working directory
static forkFrom(
  sourcePath: string,
  targetCwd: string,
  sessionDir?: string
): SessionManager
  // Creates new session in targetCwd with full history from source
  // Links parentSession to source for audit trail
  // Copy-on-write semantics (source unchanged)

// Discover sessions
static async list(
  cwd: string,
  sessionDir?: string,
  onProgress?: SessionListProgress
): Promise<SessionInfo[]>
  // List all sessions in projectdirectory
  // Sorted by modified date (newest first)
  // Returns SessionInfo {path, id, name, created, modified, messageCount, ...}

static async listAll(
  onProgress?: SessionListProgress
): Promise<SessionInfo[]>
  // List all sessions across all projects
  // Scans ~/.pi/agent/sessions/*/
```

## Practical Patterns

### Name a Session

```typescript
const manager = /* ... */;
manager.appendSessionInfo("my-research-session");
```

Stored as SessionInfoEntry, retrieved by `/resume` in TUI and session browsers. Can be updated multiple times (latest name wins).

### Add a User Annotation

```typescript
// Label a key message
manager.appendLabelChange(messageId, "pivotal insight");

// Remove label
manager.appendLabelChange(messageId, undefined);
```

Used for navigation in tree browsers (`f` to filter by labels).

### Extract a Branch to New Session

```typescript
// User explored dead-end path but wants to save just the productive thread
const newPath = manager.createBranchedSession(goodBranchLeafId);

// newPath points to fresh session file in project directory
// Original session unchanged
// Linke via parentSession for reproducibility
```

### Navigate to a Prior Point & Explore Alternatives

```typescript
// Current path: msg-1 → msg-2 → msg-3 → msg-4
// Want to try different msg-3'

manager.branch(msg2Id);  // Move leaf back to msg-2
manager.appendMessage(altMsg3);  // Appends as msg-2's child, creates fork
// Session tree now has msg-3 and msg-3' as siblings

// Later, optionally label the branch point
manager.branchWithSummary(msg2Id, "Exploring alternative approach B");
```

### Hook into Branching/Compaction (Extensions)

```typescript
// Extension can listen to session events and add metadata

agent.subscribe((event) => {
  if (event.type === "branch_summary_requested") {
    // Generate AI summary + store custom details
    const details = {artifactIndex: [...], diffs: [...]};
    manager.branchWithSummary(event.fromId, summary, details, true);
  }
});

// Later, retrieve extension data
const branchEntry = manager.getEntry(branchId);
if (branchEntry.type === "branch_summary") {
  console.log(branchEntry.details);  // Extension data intact
}
```

### Add Custom Data to LLM Context

```typescript
// Extension wants to inject structured info into conversation

manager.appendCustomMessageEntry(
  "research-tool",  // customType for filtering
  "# Research Summary\n- Topic X\n- Finding Y",
  true,             // display in TUI
  {url: "...", timestamp: "...", source: "..."}  // details not sent to LLM
);

// In LLM context, this appears as a user message
// In TUI, styled distinctly from normal user messages
// Extension can later query details via getEntry()
```

## Persistence & Lineage

### Disk Layout

```
~/.pi/agent/sessions/
├── --Users-tonyday567-project-1--/
│   ├── 2026-03-02T14-36-00-000Z_abc-123-uuid.jsonl
│   ├── 2026-03-02T15-02-10-517Z_def-456-uuid.jsonl     (forked from prev)
│   └── ...
├── --Users-tonyday567-outside--/
│   ├── 2026-03-02T02-47-02-709Z_b423510f-...-uuid.jsonl (flip)
│   └── ...
```

Directory names encode the project path (safe for filesystems). Filename pattern: `<ISO-timestamp>_<UUID>.jsonl`.

### Lineage

```
source.jsonl (project-1)
  ↓ forkFrom(source, targetCwd)
target.jsonl (project-2)
  [header has parentSession: "path/to/source.jsonl"]
  [all content copied; new ID]
```

Enables:
- Session ancestry tracking (where did this come from?)
- Reproducibility (recreate the source conversation)
- Audit trails (graph of decision branches across projects)

## Extension Patterns ⟜ Maximum Extensibility

### Detecting Branching/Compaction Intent

Pi-mono uses event hooks so extensions can:

1. **Intercept branch points** and generate AI summaries
2. **Enhance compaction** with artifact metadata
3. **Inject messages** with rich extension data
4. **Add custom entries** for stateful operations (e.g., tool-specific logs)

See: `packages/coding-agent/examples/extensions/` for working examples.

### Custom Entry Lifecycle

```typescript
// Write
const entryId = manager.appendCustomEntry("my-extension", {
  state: "analyzing",
  progress: 42
});

// Later, query
const entry = manager.getEntry(entryId);
console.log(entry.data.state);

// Or during compaction
manager.appendCompaction(summary, firstKept, tokens, {
  customEntries: [entryId],  // Extension can preserve state
  version: 1
});
```

## Designer Notes

**Immutability is intentional.** Entries never change; only append. This enables:
- Exact replay (walk the log)
- Branching without merging complexity
- Audit trails (nobody rewrites history)
- Concurrent agents (lock-free reads)

**Persistence is optional.** SessionManager can run in-memory (persist=false) for testing or sub-agents. When persisting, JSONL is append-only (cheap, no full rewrites).

**Tree structure is first-class.** parentId makes branching native, not bolted-on. Forking, branching, and compaction preserve the tree.

**Extension seams are explicit.** CustomEntry, CustomMessageEntry, hook flags (fromHook), details fields—all designed so extensions participate without ceremony.

## For Makers ⟜ Integration Checklist

- [ ] Understand SessionEntry types and tree structure (read this card)
- [ ] Spin up a SessionManager instance (for your domain)
- [ ] Implement basic append/navigate operations
- [ ] Test branching and forking
- [ ] Add custom entries for your extension state
- [ ] Hook into branch/compaction if needed
- [ ] Link to session in your tool's output/help

If you're adding a feature that touches sessions, consider:
- Does it need a new entry type? (Usually no—custom entries work)
- Does it participate in LLM context? (→ CustomMessageEntry)
- Does it need lineage tracking? (→ fork pattern + parentSession)
- Does branching affect it? (→ listen to branch events)

Let the structure guide you. Immutable logs are powerful.
